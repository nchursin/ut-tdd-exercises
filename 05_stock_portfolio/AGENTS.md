# Принципы разработки

## TDD (Test-Driven Development) по Кенту Беку

**TDD** — это методология разработки, основанная на цикле «Красный — Зелёный — Рефакторинг».

### Цикл TDD

1. **Красный** — напишите тест, который не проходит
   - Тест должен проверять одну конкретную функцию
   - Тест должен быть простым и понятным
   - На этом этапе код реализации ещё не существует

2. **Зелёный** — заставьте тест пройти
   - Напишите минимальный код, чтобы тест прошёл
   - Не бойтесь дублирования и временных решений
   - Главное — быстро получить зелёный свет

3. **Рефакторинг** — улучшите код
   - Устраните дублирование
   - Улучшите читаемость и структуру
   - Убедитесь, что все тесты продолжают проходить

### Правила TDD

- **Не пиши производственный код без теста** — это главное правило
- Пиши только необходимые тесты (тест должен компилиться)
- Пиши только необходимый код, чтобы тест прошёл
- **Никогда не опережай тесты** — если тест ещё не написан, код писать нельзя
- **Соблюдай цикл строго** — Красный → Зелёный → Рефакторинг

### Что создавать на фазе Red (красный тест)

**МОЖНО:**
- Пустые классы (без полей)
- Пустые методы (с заглушкой `throw new UnsupportedOperationException()`)
- Сигнатуры методов (название, параметры, тип возврата)

**НЕЛЬЗЯ:**
- Поля классов
- Реализацию методов
- Конструкторы с инициализацией
- Вспомогательные методы
- **Любую реализацию до написания теста**

**Пример правильного подхода на фазе Red:**
```java
// МОЖНО: пустой класс с пустым методом
public class BankAccount {
    public void deposit(int amount) {
        throw new UnsupportedOperationException("Not implemented yet");
    }
}

// НЕЛЬЗЯ: поля и реализация
public class BankAccount {
    private Money balance = Money.zero(); // НЕЛЬЗЯ
    
    public void deposit(int amount) {
        this.balance = this.balance.add(Money.of(amount)); // НЕЛЬЗЯ
    }
}
```

### Частые ошибки при работе по TDD

 **Поспешил с реализацией** — написал код до того как тест упал
**Добавил поля заранее** — поля создаются только на фазе Зелёный
**Реализовал конструктор** — инициализация только когда тест требует

**Правильный порядок:**
1. Написал тест (Красный)
2. Убедился что тест падает
3. Написал минимальный код (Зелёный)
4. Убедился что тест проходит
5. Рефакторинг

## Бережливый подход (Lean)

**Lean** — это философия, направленная на создание ценности при минимальных затратах ресурсов и времени.

### Основные принципы

1. **Определяй ценность** — с точки зрения клиента
2. **Строй поток создания ценности** — устраняй потери
3. **Создавай непрерывный поток** — без остановок и ожиданий
4. **Вытягивай ценность** — делай только то, что нужно сейчас
5. **Стремись к совершенству** — постоянное улучшение (Kaizen)

### Виды потерь (Muda)

- Частично сделанная работа
- Лишние функции
- Переключение между задачами
- Ожидания
- Лишние перемещения
- Дефекты
- Лишняя обработка

## KISS (Keep It Simple, Stupid)

**KISS** — принцип, призывающий к простоте решений.

### Суть принципа

- Простые решения лучше сложных
- Избегайте излишней сложности
- Код должен быть понятным и лёгким для поддержки
- Простота ≠ примитивность, простота = ясность

### Как применять

- Выбирайте самое простое рабочее решение
- Избегайте преждевременной оптимизации
- Делайте код читаемым
- Упрощайте архитектуру

---

## YAGNI (You Aren't Gonna Need It)

**YAGNI** — принцип, который гласит: «Вам это не понадобится».

### Суть принципа

- Не добавляйте функциональность, пока она не нужна
- Не создавайте «на будущее»
- Избегайте избыточного проектирования
- Добавляйте код только когда есть реальная потребность

### Как применять

- Реализуйте только текущие требования
- Не создавайте абстракции «про запас»
- Удаляйте неиспользуемый код
- Доверяйте рефакторингу — вы сможете добавить нужное позже

---

## Взаимосвязь принципов

Все эти принципы работают вместе:

- **TDD** помогает соблюдать **YAGNI** — вы пишете только код, который нужен для прохождения теста
- **TDD** способствует **KISS** — простой код легче тестировать
- **Lean** устраняет потери, а **YAGNI** предотвращает создание лишнего кода
- **KISS** и **YAGNI** поддерживают бережливый подход, минимизируя отходы

## Примеры минимальной реализации для первого теста

Для теста `test_add_empty_string_returns_zero()` правильная минимальная реализация:

```python
class StringCalculator:
    def add(self, input):
        return "0"
```

**Почему именно так:**
- Первый тест проверяет только один случай: пустая строка `""` должна вернуть `"0"`
- Минимальный код — просто вернуть `"0"` без каких-либо проверок
- Это следует принципу YAGNI — не добавляем проверку `if input == ""` пока тесты не потребуют
- KISS — самое простое рабочее решение

## Правило рефакторинга тестов

**Если накопилось 3 или больше похожих тестов** — на фазе рефакторинга предложи объединить их в параметризованный тест.

**Пример:**
Вместо:
```python
def test_add_empty_string_returns_zero():
    assert calculator.add("") == "0"

def test_add_single_number_returns_itself():
    assert calculator.add("1") == "1"

def test_add_two_numbers_returns_sum():
    assert calculator.add("1,2") == "3"
```

Сделать:
```python
@pytest.mark.parametrize("input,expected,description", [
    ("", "0", "empty string returns zero"),
    ("1", "1", "single number returns itself"),
    ("1,2", "3", "two numbers return sum"),
])
def test_add_various_inputs(input, expected, description):
    assert calculator.add(input) == expected, description
```

**Преимущества:**
- Уменьшает дублирование кода тестов
- Упрощает добавление новых тестовых случаев
- Делает output тестов более читаемым
- Следует принципу DRY (Don't Repeat Yourself)
